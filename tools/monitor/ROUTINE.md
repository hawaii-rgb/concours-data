# Concours 일일 대회 모니터링 루틴 (Claude 클라우드 예약용)

이 문서는 **매일 1회 실행되는 Claude 클라우드 예약 루틴(`/schedule`)의 태스크 프롬프트**다.
아래 "루틴 지시" 전체를 예약 태스크로 등록한다. 목적: 신규·회차전환·일정변경 대회를
감지해 **사용자 승인 없이 곧바로** `concours-data`에 push → 앱 아이콘 배지로 사용자에게 알림.

---

## 배경 (에이전트가 알아야 할 것)

- **데이터 리포**: `github.com/hawaii-rgb/concours-data` — 앱이 실시간 fetch하는 라이브 소스.
  - 파일: `competitions.json`(대회당 항목 1개, 최신 회차만) + `images/<slug>/pNN.png`(포스터).
  - 앱은 `officialName|round|month`를 id로 씀 → **회차·월이 바뀌면 새 id로 인식 → 앱 배지 자동 점등.**
- **스크립트**: 이 저장소 `tools/monitor/`
  - `targets.py` — 오늘의 감시 타깃 생성(연중개최예상 기반).
  - `safe_merge.py` — 변경 반영 가드레일(이것으로만 JSON을 수정한다).
- **오늘 날짜**는 루틴 실행 환경의 실제 시스템 날짜를 쓴다.

## ⛔ 절대 규칙 (안전·저작권)

1. **`competitions.json` 편집은 `safe_merge.py`로만.** 직접 손으로 JSON을 고치지 않는다.
2. **사실 필드만.** 날짜·장소·부문·접수·회차·지역·주최. (safe_merge가 그 외를 무시함)
3. **`images/` 디렉토리는 절대 건드리지 않는다.** competitions.json만 커밋. (과거 `git add -A`가
   이미지 대량 삭제를 원격에 전파한 사고가 있었음 — `git add competitions.json`만 스테이징.)
4. **포스터는 지자체 공홈 원본만.** 국악신문·국악타임즈의 **워터마크 이미지/요강 원문은 재사용 금지**
   (저작권). 이 3소스는 "대회가 떴다"는 **발견 + 사실정보 추출**용으로만 쓴다. 포스터를 못 구하면
   `noticeImages`는 빈 채로 두고 `sourceUrl`에 공홈/공고 링크만 넣는다.
5. **삭제 금지.** 신규 추가 / 기존 항목의 사실 필드 갱신만. 대회를 지우지 않는다.
6. 확신이 낮은 항목은 **넣지 않는다.** 잘못된 정보가 바로 실사용자 앱에 노출되므로 보수적으로.
   (사용자는 오류를 대개 못 잡아낸다 → **애초에 안 틀리는 게 목표.** 빠뜨리면 다음날 재시도되지만,
   틀리면 실사용자에게 바로 노출된다. 의심스러우면 반영 대신 스킵.)
7. ⛔ **하드파싱 금지.** 소스 HTML을 정규식·CSS셀렉터로 긁지 않는다. **클로드가 실제 게시글·요강·
   포스터를 풀 추론으로 읽고 사실을 추출한다** — 이미지 요강은 비전으로 OCR·판독. 사이트 구조가
   바뀌어도 견고해야 하고, 정규식 추출의 오탐(엉뚱한 회차·연도·다른 대회 혼입)을 원천 차단한다.
   (단, `targets.py`·`safe_merge.py`는 우리 competitions.json만 다루는 결정적 안전 스크립트이므로 계속 사용.)
8. ★ **검증 없이 쓰지 않는다.** 아래 "신뢰성 검증 게이트"를 **모두 통과한 사실만** changes.json에 넣는다.

## 감지 소스 3개 — **신뢰할 수 있는 사실 소스**

국악신문·국악타임즈·콘테스트코리아는 **신뢰한다.** 여기서 확인된 신규·회차전환·일정 정보를
**사실로 반영할 수 있다** — 공홈이 없거나 안 열려도(503 등) 무방하다. 공홈은 있으면 교차확인·보강용
이지 필수는 아니다. 클로드가 각 사이트를 **열어 읽고**(리스트·검색·최근 공고) 국악 경연대회를 후보로
뽑는다 — 정규식 스크래핑 금지, 네가 직접 판독. (콘테스트코리아는 회차 반영이 지연될 수 있으니 회차
숫자는 게시글 원문으로 확인.)

- **콘테스트코리아** `contestkorea.com` — "국악" 검색 또는 음악/공연 분야 리스트를 클로드가 훑어
  최근 국악 경연대회 게시글을 찾는다. 상세 페이지를 열어 내용을 읽는다.
- **국악신문** `kukak21.com` — 최근 기사·공지에서 대회 공고를 읽는다.
- **국악타임즈** `gugaktimes.com` — 최근 기사에서 대회 공고를 읽는다.

→ 후보를 찾으면 아래 **검증 게이트**로 (날짜·회차의) 정합성을 확인한 뒤 반영한다.

## 네이버 보조 검색 (지자체 공홈·포스터·누락분)

`https://search.naver.com/search.naver?query=<대회명>` 페치. HTML에 go.kr/or.kr 공홈 링크가
박혀 있음(grep 추출 가능). **미국 기반 WebSearch는 쓰지 말 것**(한국 지자체 공홈 누락).
대회명으로 검색해 ① 공식 홈페이지 공고(1순위 사실 소스) ② 공홈 포스터 원본을 확인한다.

## ★ 신뢰성 검증 게이트 (반영 전 필수 — 하나라도 걸리면 그 항목 스킵)

각 후보 사실(날짜·회차·장소 등)은 아래를 **모두** 통과해야만 changes.json에 넣는다. 사용자가 오류를
못 잡으므로, "확실치 않으면 넣지 않는다"가 기본값이다.

1. **소스 확정 (3소스 신뢰).** 날짜·회차·장소·접수는 국악신문·국악타임즈·콘테스트코리아 중
   **한 곳 이상에서 확인되면 반영**한다. 공홈(지자체·문화재단)이 열리면 교차확인해 더 정확히 —
   하지만 **공홈이 없거나 503 등으로 안 열려도 3소스 정보로 반영한다**(공홈은 필수가 아님).
   소스 어디에도 정보가 없어 확인 자체가 안 되는 것만 스킵.
2. **요일 정합성 (연도 오류 잡는 결정타).** 공고가 요일을 명시하면(예: "7. 11.(토)"), 그 날짜의 **실제
   요일을 계산해 일치하는지** 확인한다. 불일치 = 연도/날짜 오류 → 스킵·재확인.
   확인법: `python3 -c "import datetime;print(datetime.date(2026,7,11).strftime('%a'))"` (Sat=토).
   (실측 예: 2026-07-11=토, 2025-07-11=금 → 공고가 "(토)"면 2026이 맞다.)
3. **회차↔연도 정합성.** 연 1회 대회면 제N회=YYYY, 제(N+1)회=YYYY+1. 새 회차는 데이터셋 최신 회차의
   **+1**이어야 하고 연도가 주기와 맞아야 한다. 모순(예: 데이터 제25회=2026인데 "제26회=2026")이면
   플래그·스킵. **회차 건너뜀·연도 역행**은 의심 신호.
4. **공고 시점 상식.** 공고 발행일이 개최일보다 앞서고(보통 1~3개월 전), 개최일이 오늘 기준 과도한
   과거/미래가 아닌지. (발행일은 게시글에서 클로드가 읽어 확인.)
5. **단일 소스도 신뢰.** 3소스(국악신문·국악타임즈·콘테스트코리아) 중 한 곳에서만 확인돼도 반영한다
   (교차확인 필수 아님). 공홈이 함께 확인되면 더 좋다.
   - ★ **단, 공지는 종종 바뀐다** — 개최일·장소 변경, 접수 연기, 요강 수정 등. 한 번 기록했다고
     고정으로 보지 말 것. **기존 항목도 매일 재확인**해 변경이 있으면 갱신하고, **항상 최신 공고
     상태**를 사실로 삼는다(소스끼리 과거·현재가 다르면 최신이 우선).
6. **확신 게이트.** 위 중 하나라도 불확실 → **넣지 않는다.** (오탐을 넣는 비용 ≫ 하루 늦게 잡는 비용)

> 위 검증은 클로드가 **읽고 추론해서** 수행한다(요일 계산만 파이썬 한 줄 사용). 정규식으로 페이지에서
> 숫자를 긁어 그대로 쓰는 방식은 금지 — 그게 바로 회차·연도 오탐의 원인이다.

## 루틴 지시 (매일 실행)

1. **준비**: 이 저장소(concours-data)에서 실행된다. 스크립트는 tools/monitor/ 에 있다.
2. **타깃 생성**:
   `python3 tools/monitor/targets.py --file competitions.json > targets.json`
   → `targets[]`(곧 개최 예정이라 새 회차가 뜰 만한 대회) + `all_keys`(전체 정규화 키, 신규 판별용).
3. **스캔**: 3소스 + 네이버로 아래를 찾는다.
   - **(a) 회차 전환**: `targets[]`의 각 대회에 대해, 마지막으로 알려진 `lastRound`보다 **새 회차**가
     공고됐는가? (예: lastRound=51 → 제52회 공고 발견) → `update` 후보.
   - **(b) 일정 변경**: 기존 대회의 접수마감·대회일이 바뀌었는가? → `update` 후보.
   - **(c) 신규 대회**: 정규화 이름이 `all_keys`에 **없는** 대회가 공고됐는가? → `new` 후보.
   - 정규화 규칙: 회차·연도·괄호·공백 제거(`targets.py`의 base와 동일). `update`의 `match_base`에는
     반드시 `targets[].base` 값을 그대로 쓴다(직접 만들지 말 것).
4. **사실 확정 (소스 원문을 클로드가 직접 읽어서)**: 각 후보의 사실은 3소스 게시글을 열어(가능하면
   네이버로 공홈도 찾아 교차확인) 클로드가 **본문·요강·포스터를 풀 추론으로 판독**(이미지면 비전 OCR)해
   채운다:
   날짜(`competitionDate`), 접수기간(`applyStart`/`applyEnd`), 장소(`venue`), 부문(`genres`), 부별(`levels`),
   지역(`region`), 주최(`host`), 회차(`round`), 월(`month`=YYYY-MM), 공고 링크(`sourceUrl`).
   - ★ 각 항목은 위 **신뢰성 검증 게이트를 통과해야만** 채택 — 3소스 신뢰(게이트 1·5)이되, 날짜·회차의
     **내부 정합성**(요일·회차↔연도·공고시점)은 반드시 확인. 정합성 모순이면 스킵.
   - **확인 안 된 필드는 비운다(추측 금지).** 정규식으로 긁은 값 그대로 쓰지 않는다.
   - `sourceUrl`은 확인한 원문 링크(3소스 게시글 또는 공홈). 포스터는 공홈 원본만 —
     이 루틴에선 `noticeImages`를 비워 둔다(포스터 수집은 별도 트랙).
5. **변경 작성**: 위 후보들을 `changes.json`으로 만든다(형식은 `safe_merge.py` 상단 주석 참조).
6. **안전 반영**:
   `python3 tools/monitor/safe_merge.py --file competitions.json --changes changes.json`
   → 요약과 `COMMIT_MSG:` 한 줄이 출력된다. (변경 0이면 여기서 종료 — 커밋하지 않음)
7. **커밋 & push** (concours-data에서):
   - `git add competitions.json` ← **오직 이 파일만.** (`git add -A`/`git add .` 금지)
   - `git status`로 `images/` 삭제(`D images/...`)나 의도치 않은 변경이 없는지 확인. 있으면 **중단**.
   - `git commit -m "<COMMIT_MSG 내용>"` → `git push origin main`.
   - `.bak.monitor*` 백업 파일은 커밋하지 않는다(스테이징에서 제외).
8. **보고**: 그날 무엇을 추가/갱신했는지(safe_merge 요약)를 실행 로그로 남긴다. 변경 없으면 "변경 없음".

## 실패·불확실 처리
- 소스가 안 열리거나 애매하면 **그 항목을 건너뛴다**(부분 실패 허용). 다음 날 다시 시도됨.
- 한 곳에서만 보이고 교차확인이 안 되는 신규는 보수적으로 보류하거나, sourceUrl에 발견 링크만 달아
  최소 정보로 넣는다(뒤에 사용자가 정정).

## 사용자용 등록 안내 (한 번만)
- 이 문서의 "루틴 지시"를 Claude Code `/schedule`(또는 클라우드 예약 에이전트)에 **매일 1회**로 등록.
- 예약 에이전트가 이 저장소(concours-data)에 접근·푸시 가능해야 함.
- 오류 발견 시: 세션에서 "○○ 대회 정보 틀렸어" 식으로 알려주면 데이터 정정·push(또는 해당 커밋 revert).
